# PRD：v4-growth-evidence-chain

## 1. 背景

v3 已经把个人主页升级为“精选项目卡片 + Workbench 地图”，解决了“项目太少、证据不够”的问题。但复盘后发现一个新的叙事问题：页面容易让访问者把注意力放在工具、项目和工作台上，而不是放在“这个人为什么值得被重新认识”上。

本轮不是否定 v3，而是调整叙事层级：项目继续保留，但项目是证据，不是主角。主页的主角应该是一个正在用 AI、结构化表达和持续实践快速成长的人。

## 2. 本次目标

把首页从“AI 工作台 / 项目地图”重构为“成长证据链”。

成功后访问者应该能在 30 秒内看懂：

- 这不是一个工具陈列页，而是一个人的成长展示页。
- 用户正在把想法持续变成可打开、可验证、可复用的作品。
- 项目、Build Log、能力地图和联系方式都在证明同一件事：高斜率、持续探索、持续产出。
- 即使不懂 Codex、Agent、Workbench，也能理解这个人有行动力和潜力。

## 3. 范围

### 本次做

- 重写 Hero 的叙事重心：从“AI 工作台”转为“我如何用 AI 把想法变成作品”。
- 调整“我是谁”区块：突出人、成长斜率、表达与执行，而不是工具集合。
- 重构成长线路：从工具流程改为成长证据链，例如“想法 -> 结构化 -> 做成作品 -> 公开展示 -> 继续迭代”。
- 调整项目卡片结构：每张卡片都要回答“问题 / 我做了什么 / 它证明什么”。
- 保留 v3 已筛选通过的项目，但降低项目地图的主角感。
- 将 Workbench 区块改写为“作品背后的方法 / 工作方式”，或在页面顺序上放到项目证据之后。
- 更新能力地图和 Build Log，让它们支撑“成长证据链”而不是重复罗列工具。
- 保持单页静态站架构，不新增后端、不新增 CMS、不改部署平台。

### 本次不做

- 不新增更多项目数量。
- 不恢复已被用户否定的项目或概念。
- 不公开私密路径、聊天原文、账号 ID、内部数据或未确认链接。
- 不新增项目详情页。
- 不改技术栈、路由结构或部署方式。
- 不替用户 commit / push。

## 4. 信息架构

页面仍是单页结构，但叙事顺序调整为：

1. **Hero：先讲人和成长**
   - 用一句强标题说明：用户正在用 AI 把想法变成真实作品。
   - 副标题解释：这是一个持续迭代的个人主页，也是 SpecDrivenCoding 的作品证明。
   - CTA 指向项目证据和联系方式。

2. **Identity：建立定位**
   - 回答“我是谁”：一个高斜率、持续把想法结构化并做成作品的人。
   - 只保留必要的工具词，避免工具压过人的判断。

3. **Growth Evidence Chain：成长证据链**
   - 用少量阶段展示从想法到作品的转换过程。
   - 每个阶段都要有“行为 + 结果 + 证明的能力”。

4. **Projects：项目作为证据**
   - 保留已确认的项目集合。
   - 项目卡片从“项目介绍”改成“证据卡片”：
     - 这个项目解决什么问题？
     - 我做了什么？
     - 它证明我有什么能力？

5. **Method / Workbench：作品背后的方法**
   - Workbench 不再是主叙事。
   - 用来解释这些作品背后的工作方式：结构化、Agent 协作、内容生产、视觉交付、持续迭代。

6. **Abilities：能力地图**
   - 能力必须落回证据，不做空泛标签。
   - 每项能力最好能指向具体项目或行为。

7. **Build Log：持续构建证明**
   - 保留热力图和最近更新。
   - 文案强调“持续产出和迭代”，不是技术流水账。

8. **Contact：行动入口**
   - 保持 GitHub、Gmail、电话、微信等真实入口。
   - 语气自然，面向老师、朋友、家人和老板都能接受。

## 5. 内容原则

- 说人话，少用只有 AI 圈能懂的术语。
- 先给判断，再给证据。
- 项目文案不要像功能清单，要像能力证明。
- “AI / Codex / Agent / Workbench”可以出现，但不能抢走主角。
- 不把未完成、不可验证、对外价值不清楚的碎片放到主叙事里。

## 6. 已确认保留与删除

### 保留

- 个人主页
- 个人公众号
- 微信 ClawBot
- AI 内参
- 思想孵化
- 桌面助手原型
- PPT / PDF 视觉交付

### 不保留

- LoSquare
- Codexbox Skill 工作栏
- 中横分析法 / 纵横分析法相关表述
- “蚂蚁学姐”这个名字

## 7. 验收标准

- 首屏不再让人误以为这是工具台或项目目录。
- 页面主线能被概括为“一个高斜率的人如何持续把想法变成作品”。
- 项目卡片能清楚回答“证明了什么”，不是只介绍“做了什么”。
- Workbench / 方法区服务于作品证据，不抢主线。
- 页面不出现已否定项目和不可公开信息。
- 桌面和移动端可读，没有明显文字溢出、遮挡或横向裁切。
- 本地构建通过；如构建工具异常，需记录异常原因和临时预览方式。

## 8. 风险与待解决问题

- 风险 1：叙事太抽象，导致页面像个人宣言而不是作品证据。处理方式：每个抽象判断后面必须接真实项目或构建行为。
- 风险 2：删除工具感后，技术可信度下降。处理方式：GitHub、Build Log 和项目证据保留，但排序上服从人的叙事。
- 风险 3：项目卡片信息量变大，移动端容易拥挤。处理方式：卡片字段要短，必要时用折叠层级或紧凑布局。
- 风险 4：本地构建曾出现过工具链卡住的情况。处理方式：实施前先确认构建状态；若仍异常，先修构建或明确临时预览流程。
