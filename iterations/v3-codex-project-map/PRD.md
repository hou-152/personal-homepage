# PRD：v3-codex-project-map

## 1. 背景

当前个人主页已经完成 v1 页面、真实联系方式、GitHub 仓库和 GitHub Pages 上线。现有项目区只有少量“主页本身”的过程证据，无法充分展示用户在 Codex 里长期积累的项目、工作流和 Agent 基建。

本轮迭代要把 Codex 项目体系放进网站，但不能把本机文件夹、私有路径、聊天内容或内部工具细节原样公开。页面应该把这些项目翻译成普通访问者能理解的作品证据。

## 2. 本次目标

把主页的项目展示升级为“精选项目卡片 + Codex Workbench 地图”。

成功后访问者应该能看懂：

- 用户不只是做了一个个人主页，而是在持续搭建 AI 协作工作台。
- 这些项目覆盖内容生产、个人公众号、微信 Agent、思想孵化、桌面助手原型、PPT/PDF 视觉交付和个人主页。
- 每个项目都能证明一种可迁移能力，而不是只展示文件名或工具名。
- 私密项目只展示公开安全的抽象成果，不泄露个人数据。

## 3. 范围

### 本次做

- 重写 `src/data/projects.ts`，加入 7 个精选项目卡片。
- 新增 Codex Workbench 数据，用分组方式展示项目版图。
- 在页面中新增 Workbench 区块。
- 更新能力地图、成长线路、Build Log，让它们和真实 Codex 项目一致。
- 保持当前静态站架构，不新增后端、不接 CMS。
- 本地构建验证通过。

### 本次不做

- 不公开本机路径、私有仓库、聊天原文、Notion/Readwise ID。
- 不把所有历史目录逐个做成卡片。
- 不新增项目详情页。
- 不新增外部数据抓取或 GitHub 自动同步。
- 不改部署平台。

## 4. 信息架构

页面仍保持单页结构，但项目相关内容升级：

1. Hero：强调“把 AI 协作工作台变成可见作品”。
2. 我是谁：表达当前方向是 AI 协作、个人 Agent、知识系统和公开展示。
3. 成长线路：从 SpecDriven 到 Workbench，再到公开作品集。
4. 项目卡片：精选 7 个最能证明能力的项目。
5. Codex Workbench：按工作流方向展示更完整的项目版图。
6. 能力地图：用项目结果证明能力。
7. Build Log / 最近更新：记录本轮 v3 项目地图迭代。
8. 联系入口：保持 GitHub、Gmail、电话、微信。

## 5. 项目选择原则

### 放入项目卡片

满足任意两条：

- 能被普通访客理解为一个作品或系统。
- 能证明用户的成长、执行或 AI 协作能力。
- 有明确结果，而不是只停留在想法。
- 公开表达时不依赖私密数据。

### 放入 Workbench 地图

适合表达方向和体系，但不适合展开细节的项目进入 Workbench 地图。例如私密数据较多、仍在内部试验、或由多个小工具组成的工作流。

### 不展示

- 私人聊天原文。
- 本机绝对路径。
- 账号、token、database id、Reader/Notion 页面 id。
- 低完成度、对外价值不清楚的碎片。

## 6. 验收标准

- 首页项目区至少有 7 个项目卡片。
- 页面包含 Codex Workbench 分组区块。
- 所有展示文案对非技术访客可读。
- GitHub 链接只指向公开安全入口。
- 页面不出现本机路径、私有 ID、密钥或内部配置。
- `npm run build` 通过。
- 移动端布局不出现明显文字溢出或卡片挤压。
